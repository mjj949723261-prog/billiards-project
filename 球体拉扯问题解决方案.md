# 台球游戏球体拉扯问题解决方案

## 问题诊断

### 核心问题
击球初期多球碰撞时出现"拉扯"现象（球体运动不自然、抖动或穿透），这是三层问题叠加的结果：

1. **本地物理不够稳定**
   - 物理更新按帧推进而非固定时间步长
   - 不同帧率下轨迹会自然分叉
   - `Ball.update()` 未使用真实 `dt` 参数

2. **多球密集碰撞解算太粗**
   - 离散碰撞检测（帧后检测，非连续检测）
   - 单轮位置修正，无法处理多球链式约束
   - 修正系数 `correctionScale = 0.52` 是经验值，不是稳定求解器

3. **运动中网络同步持续改写球状态**
   - 运动期整桌 `isLive: true` 状态持续发送
   - 接收端对位置和速度做混合（lerp）
   - 服务端每 100ms 推送 `SYNC_STATE`
   - 形成闭环：本地和远端互相拉扯对方的中间态

### 为什么击球初期最明显
开球初期同时触发所有弱点：
- 球速高
- 接触对数多
- 连锁碰撞密集
- 重叠修正频繁
- 双端模拟误差增长最快
- live 同步写回最频繁

## 解决方案（六阶段演进）

### 第一阶段：止血 - 切断运动期 live 闭环

**目标**：先解决当前最明显的"击球初期多球碰撞拉扯"

**动作**：
1. 停止运动期整桌 `isLive` 状态持续写回物理层
2. 击球方不再接收/应用来自远端的运动期球桌纠偏
3. 服务端不再把运动中的 `ballState` 当成房间权威快照循环广播
4. 保留业务同步：
   - `SHOOT` 击球指令
   - `AIM` 瞄准数据
   - 白球摆放（低速 UI 交互，保持临场感）
   - 停球后结果同步
   - 回合/timer/score/group 状态

**核心原则**：切断网络对物理层的高频写入

**涉及文件**：
- `daily-billiards-vanilla_web/src/game.js:493` - 停止运动期 sendSync
- `daily-billiards-vanilla_web/src/game.js:323` - 修改 applyGameStateSnapshot 逻辑
- `billiards-server/src/main/java/com/billiards/game/service/RoomService.java:172` - 收缩 SYNC_STATE 职责
- `billiards-server/src/main/java/com/billiards/game/service/RoomService.java:226` - 不再广播运动中 ballState

---

### 第二阶段：引入视觉层与物理层分离

**目标**：把残余误差和收敛过程隔离到视觉层，避免肉眼看到抖动和跳变

**动作**：
1. 为每颗球建立双状态系统：
   - `physicsPos / physicsVel` - 物理层
   - `renderPos / renderRot` - 渲染层
2. 碰撞检测、规则判定、入袋判断只基于 `physicsPos`
3. 渲染层每帧让 `renderPos` 平滑追赶目标状态
4. 非击球方的球桌展示以快照缓冲与插值为主

**核心原则**：视觉层和物理层分离，避免把正常的小误差放大成用户肉眼可见的抖动

**涉及文件**：
- `daily-billiards-vanilla_web/src/entities/ball.js` - 添加 renderPos/renderRot 属性
- `daily-billiards-vanilla_web/src/game.js` - 渲染逻辑使用 renderPos
- `daily-billiards-vanilla_web/src/core/physics.js` - 物理计算只用 physicsPos

---

### 第三阶段：固定步长物理（核心基石）

**目标**：把本地物理从"按帧行为"改成"按时间行为"

**动作**：
1. `game.update()` 引入 `accumulator` 机制
2. 物理以固定 `fixedStep`（推荐 1/120s）跑多个 substep
3. `Ball.update(dt)` 使用真实步长参数
4. 摩擦、减速、停止阈值全部改为 dt 相关公式
5. `Date.now()` 只留给 UI、倒计时、特效，不再直接影响球路

**核心原则**：消除不同设备刷新率（60Hz vs 144Hz）带来的轨迹发散问题

**涉及文件**：
- `daily-billiards-vanilla_web/src/game.js:469` - 引入 accumulator 和 fixedStep
- `daily-billiards-vanilla_web/src/entities/ball.js:136` - 修改 update(dt) 签名和实现
- `daily-billiards-vanilla_web/src/entities/ball.js:139` - pos += vel * dtScale
- `daily-billiards-vanilla_web/src/entities/ball.js:152` - 摩擦改为时间相关
- `daily-billiards-vanilla_web/src/entities/ball.js:155` - 线性减速改为时间相关
- `daily-billiards-vanilla_web/src/core/physics.js:28` - updateGamePhysics 真正使用 dt

---

### 第四阶段：多球碰撞迭代求解

**目标**：解决开球堆散阶段的约束不收敛问题

**动作**：
1. 每个物理步内做 4-8 轮碰撞迭代
2. 保持稳定的碰撞遍历顺序（避免随机性）
3. 重构重叠修正策略，不再依赖单个 `correctionScale`
4. 引入 slop/bias 机制（允许微小穿透以换取稳定性）

**核心原则**：让多球堆中的约束有收敛过程，而不是一帧只碰一下就算完

**涉及文件**：
- `daily-billiards-vanilla_web/src/core/physics.js:190` - handleBallCollisions 改为迭代式
- `daily-billiards-vanilla_web/src/core/physics.js:195` - 碰撞检测逻辑
- `daily-billiards-vanilla_web/src/core/physics.js:198` - 位置修正策略重写
- `daily-billiards-vanilla_web/src/core/physics.js:224` - 冲量计算优化

---

### 第五阶段：停球后的软性对账

**目标**：避免停球时硬跳，保证最终一致性

**动作**：
1. 停球后接收最终权威快照
2. 根据误差大小分层处理：
   - **小误差**：不改 `physicsPos`，只让 `renderPos` 在 150-300ms 内平滑追上
   - **中误差**：球已判停且无碰撞风险时，温和修正 `physicsPos`，同时 `renderPos` 软追
   - **大误差**：接受显式跳转（低频场景）

**核心原则**：把第一阶段的"切掉 live 闭环"补完整，消除终点跳变

**涉及文件**：
- `daily-billiards-vanilla_web/src/game.js` - 停球后对账逻辑
- `daily-billiards-vanilla_web/src/network/game-client.js` - SYNC_STATE 接收处理分层

## 技术共识

### 四个关键争议点的最终立场

1. **第一优先级是否是"先停运动期 live 闭环"？**
   - ✅ **是，必须第一优先执行**
   - 成本最低、见效最快的"止痛药"

2. **固定步长和碰撞迭代哪个先做？**
   - ✅ **必须先做固定步长**
   - 时间轴不稳定，空间解算再多次迭代也徒劳

3. **服务端 SYNC_STATE 是否也要收缩职责？**
   - ✅ **是，强烈建议收缩**
   - 转型为"权威对账单"和"回合状态机"
   - 只广播：回合切换、AIM、SHOOT、停球后最终位置

4. **是否保留"白球摆放 live"？**
   - ✅ **必须保留**
   - 低速 UI 交互，保持双人对战临场感

### 角色分工

- **击球方**：本地实时物理，自己是主视角
- **非击球方**：快照缓冲 + 插值播放为主，物理只做辅助，不做强纠偏源
- **服务端**：状态裁决 / 快照分发 / 回合同步，不做逐步一致模拟

## 风险评估与应对

### 风险 A：运动期停止同步导致的"终点跳变"
- **应对**：第二阶段视觉物理分离 + 第五阶段软性对账
- **原则**：停止运动期整桌同步 ≠ 停止收敛机制

### 风险 B：服务器权威的实现成本
- **应对**：短中期不走"Java 端重跑与 JS 端逐步完全一致"
- **原则**：服务端先做状态裁决，不做逐步一致模拟

### 风险 C：缺乏平滑缓冲区
- **应对**：第二阶段引入 renderPos 缓冲层
- **原则**：网络驱动视觉状态，不直接写物理状态
