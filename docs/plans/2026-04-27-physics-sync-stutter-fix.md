# 台球物理碰撞“拉扯”感优化方案 (2026-04-27)

## 问题分析
当前游戏中多球碰撞时的“拉扯”感主要源于以下三个缺陷：
1. **非确定性物理**：物理引擎逻辑与渲染帧率耦合，未处理 deltaTime，导致不同客户端模拟轨迹迅速发散。
2. **缺乏时间基准**：同步消息无时间戳，客户端收到旧快照后直接应用，导致球体被向后拉回。
3. **生硬的纠偏算法**：直接使用加权平均（Lerp）修改物理位置，没有插值缓冲区和速度补偿。

## 最终解决方案

### 1. 物理层：强制固定步长 (Fixed Timestep)
- **目标**：确保物理模拟的确定性。
- **实施**：
    - 在 `game.js` 的循环中引入 `accumulator` 机制：
      ```javascript
      // 伪代码实现
      const fixedDeltaTime = 1 / 60;
      accumulator += frameDeltaTime;
      while (accumulator >= fixedDeltaTime) {
          physics.update(fixedDeltaTime);
          accumulator -= fixedDeltaTime;
      }
      ```
    - 修改 `ball.js` 的 `update` 逻辑，消除硬编码的常数位移，改为基于步长的计算。

### 2. 同步层：快照插值 (Snapshot Interpolation)
- **目标**：消除网络抖动带来的视觉跳变。
- **实施**：
    - **协议改进**：在 `SYNC_STATE` 消息中增加 `timestamp` (服务器毫秒数) 和 `sequenceId`。
    - **插值缓冲区**：非击球方客户端维护一个存储最近 200ms 快照的队列。
    - **平滑渲染**：渲染时寻找 `t = currentServerTime - delay` 所在的两个快照点，在它们之间进行线性插值，而非追赶最新的不稳定状态。

### 3. 渲染层：平滑收敛与分离
- **目标**：将物理层坐标与渲染层坐标分离。
- **实施**：
    - 为 `Ball` 类增加 `renderPos` 和 `renderRotMat`。
    - 渲染器 (`pixi-renderer.js`) 仅读取 `renderPos`。
    - 当服务器状态与本地物理偏差较大时，物理坐标直接重置，但渲染坐标通过 `Lerp` 逐渐平滑追赶到物理坐标，消除瞬移感。

### 4. 关键细节处理：静止一致性
- **状态位同步**：同步消息中包含 `isSleeping` 标记。
- **收敛算法**：当球即将停止且服务器标记为 `isSleeping` 时，客户端若距离目标位置小于 2px，则在 3 帧内加速收敛至目标坐标，并锁定位置，防止无穷尽的微小抖动。

## 预期效果
- 开局多球碰撞时，非击球方视角下球体滚动丝滑，无往复拉扯现象。
- 不同性能设备间的物理结果一致性显著提升。
- 网络波动（Jitter）在 150ms 以内时，玩家感知不到明显的跳变。
