# 击球拉扯 (Ball Jitter/Pulling) 问题根因分析报告

**日期**：2026-04-29
**状态**：已诊断
**文档编号**：REVIEW-2026-04-29-001

## 1. 问题概述
- **问题描述**：在击球过程中，尤其是多球碰撞或网络同步纠偏时，球体表现出明显的“拉扯感”。表现为球体在滚动过程中出现微小的位置往复、视觉上的畸变，以及在停球阶段突然发生的平滑位移（Rubber-banding）。
- **影响范围**：
    - **单人模式**：表现为高速运动时的视觉颤抖和旋转畸变。
    - **多人模式**：表现为网络对账导致的远程球体“滑行”纠偏。
- **严重程度评级**：**高 (High)**。直接影响台球游戏的物理真实感和竞技公平性。

**评分依据**：虽然建立了物理与渲染分离的架构，并引入了固定步长累加器，但由于关键数学逻辑（如矩阵插值）的错误、碰撞迭代中冲量处理的不完整，以及纠偏策略过于松散，导致架构优势未能转化为稳定的视觉体验。

## 3. 根因分析 (Root Cause Analysis)

### RC1：累加器时间丢失 (物理层)
- **代码位置**：`daily-billiards-vanilla_web/src/core/physics.js` L67-69
- **问题描述**：当渲染帧耗时超过 `MAX_SUBSTEPS * FIXED_TIMESTEP_MS`（约 41.6ms）时，累加器会被强制清零。
- **技术原理**：硬性丢弃未模拟的时间片，导致物理模拟时间慢于真实时间。
- **表现症状**：在低帧率或系统卡顿时，物理轨迹出现“时间跳跃”，诱发严重的网络对账偏差。
- **优先级**：P1

### RC2：碰撞求解不完整 (物理层)
- **代码位置**：`daily-billiards-vanilla_web/src/core/physics.js` L208-269
- **问题描述**：冲量（Impulse）计算仅在第 0 轮迭代执行，后续迭代仅进行位置修正。
- **技术原理**：位置纠偏（Position Correction）而不同步更新速度，导致动量守恒在密集多体碰撞中失效。
- **表现症状**：多球开球时，球体表现得像“软体”，随后产生瞬间的速度补偿抖动。
- **优先级**：P0

### RC3：数学错误的插值 (渲染层)
- **代码位置**：`daily-billiards-vanilla_web/src/entities/ball.js` L203-205
- **问题描述**：直接对 3x3 旋转矩阵的 9 个分量进行线性加权。
- **技术原理**：线性插值会破坏旋转矩阵的正交性（Orthogonality），导致渲染出的球体在旋转时产生缩放或形状扭曲。
- **表现症状**：球体在高速滚动时看起来在剧烈颤抖或尺寸微变。
- **优先级**：P0

### RC4：纠偏阈值过松 (同步层)
- **代码位置**：`daily-billiards-vanilla_web/src/game.js` L429
- **问题描述**：允许最大 28 像素（`2 * BALL_RADIUS`）的偏差而不强制重置。
- **技术原理**：在巨大偏差下仍尝试平滑追赶，导致球体看起来在缓慢“漂移”到正确位置。
- **表现症状**：典型的“拉扯”感，球体在停稳前有一个明显的滑行过程。
- **优先级**：P1

### RC5：运动期实时同步被忽略 (同步层)
- **代码位置**：`daily-billiards-vanilla_web/src/game.js` L402
- **问题描述**：运动状态下的 `live-sync` 消息被大部分过滤。
- **技术原理**：本地模拟误差在碰撞中被放大，且缺乏中间状态的持续纠正。
- **表现症状**：不同客户端看到的碰撞顺序可能不一致，最终停球点发生跳变。
- **优先级**：P1

## 4. 改进建议

### Phase 1：物理层确定性修复 (P0)
- **具体修改点**：
    1.  重构 `handleBallCollisions`，将冲量求解移入迭代循环，确保速度与位置同步收敛。
    2.  增加 `physicsAccumulatorMs` 的保护机制，避免硬性清零，改用 `Math.min` 限制单帧最大步数但不丢弃余数。
- **参数调整**：`MAX_SUBSTEPS` 提升至 12；`POSITION_CORRECTION` 降至 0.6 以平衡稳定性。
- **预期效果**：多球碰撞不再有“软绵感”，各端模拟轨迹高度一致。

### Phase 2：渲染层平滑度修复 (P0)
- **具体修改点**：
    1.  **旋转重构**：引入四元数（Quaternion）进行 `Slerp` 插值，或在矩阵插值后执行 Gram-Schmidt 正交化归一化。
    2.  **时间无关插值**：将 `smoothFactor` 与实际 `deltaTime` 绑定。
- **预期效果**：彻底消除高速滚动时的视觉抖动和形变。

### Phase 3：同步层收敛优化 (P1)
- **具体修改点**：
    1.  收紧 `shouldSnapRender` 阈值至 `BALL_RADIUS * 0.5`（7 像素）。
    2.  引入插值缓冲区（Interpolation Buffer），对远程球体应用 100ms 的延迟插值渲染。
- **风险评估**：可能会在极差网络下增加视觉延迟，但能保证运动的连贯性。

## 5. 验证方案

### 调试日志插入点
- **Physics**：在累加器溢出时输出 `Lost simulation time: ${ms}ms`。
- **Sync**：在触发纠偏时输出 `Correction applied: deviation=${val}px`。

### 测试场景
1.  **极端压力测试**：在 15fps 环境下进行强力开球，检查球体是否穿透或轨迹剧烈发散。
2.  **网络抖动测试**：模拟 150ms Latency + 50ms Jitter，观察远程球体是否仍有长距离“拉扯”。

### 成功标准
- 多体碰撞后，主从端球体最终停止位置偏差 < 1.0 像素。
- 视觉上球体在任何速度下均保持完美的正球体形状且无抖动。
